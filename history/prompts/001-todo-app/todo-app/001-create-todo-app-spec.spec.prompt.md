## **1. Purpose**

This specification defines all technical and functional requirements for building the **Phase I** in-memory, command-line Todo application using **Spec-Driven Development** with Claude Code and Spec-Kit Plus.

---

# ## **2. Scope**

This phase includes:

* A fully working terminal-based Todo application
* In-memory task management
* Implementation of all 5 Basic Level features
* Clean Python architecture under `/src`
* No database
* No external storage
* No external frameworks (only Python standard library)

---

# ## **3. Functional Requirements**

### ### **3.1 Add Task**

**Description:**
Create a new task and store it in memory.

**Requirements:**

1. Each task must include:

   * `id`: auto-incrementing integer
   * `title`: required, 1‚Äì200 characters
   * `description`: optional string
   * `completed`: boolean (default `False`)
   * `created_at`: datetime
2. Duplicate titles must be allowed.
3. Title validation must be enforced:

   * Cannot be empty
   * Cannot exceed 200 characters
4. After creation, success confirmation must be shown.

---

### ### **3.2 View All Tasks**

**Description:**
Display the full list of tasks in the console.

**Requirements:**

1. Tasks must display in a clean tabular or list format.
2. Each displayed task must show:

   * ID
   * Title
   * Completed status:

     * Completed = `‚úì`
     * Not completed = `‚úó`
   * Created timestamp
3. If no tasks exist, show:
   `"No tasks found."`

---

### ### **3.3 Update Task**

**Description:**
Update an existing task's title and/or description.

**Requirements:**

1. Task must be located using `id`.
2. If task does not exist ‚Üí show error.
3. Title remains required and must pass validation.
4. Description is optional.
5. After update, show success message.

---

### ### **3.4 Delete Task**

**Description:**
Remove an existing task using its ID.

**Requirements:**

1. Task must be identified using numeric ID.
2. If task does not exist ‚Üí show error.
3. After deletion, show success message.

---

### ### **3.5 Toggle Completion Status**

**Description:**
Mark a task as complete or incomplete.

**Requirements:**

1. Task must be located using `id`.
2. If task exists:

   * If completed ‚Üí set to incomplete
   * If incomplete ‚Üí set to complete
3. Show a confirmation message:

   * `"Task marked as completed."`
   * `"Task marked as incomplete."`

---

# ## **4. Data Model Requirements**

The Task model **must** include:

| Field         | Type     | Required | Description              |
| ------------- | -------- | -------- | ------------------------ |
| `id`          | int      | Yes      | Auto-increment unique ID |
| `title`       | str      | Yes      | 1‚Äì200 chars              |
| `description` | str      | No       | Optional text            |
| `completed`   | bool     | Yes      | Default False            |
| `created_at`  | datetime | Yes      | Auto timestamp           |

---

# ## **5. In-Memory Storage Requirements**

### Storage rules:

1. Tasks must be stored in an in-memory list or dictionary.
2. Allowed structures:

   * `list[Task]`
   * `dict[int, Task]`
3. No files, JSON, pickle, or database.
4. Data resets every time the program is restarted.

Example (allowed):

```python
tasks = []
```

---

# ## **6. Console Application Requirements**

### 6.1 Main Menu

Application must show a menu similar to:

```
1. Add Task
2. View Tasks
3. Update Task
4. Delete Task
5. Mark Task Complete/Incomplete
0. Exit
```

### 6.2 Input Validation

* Menu input must be validated.
* Invalid input must not crash the program.
* Proper error messages required.

---

# ## **7. Architecture Requirements**

The code generated by Claude Code must follow this structure:

```
/src/
    main.py            # CLI entry point
    models.py          # Task data model
    services.py        # All task CRUD operations
    utils.py           # Helper functions (optional)
```

### Rules:

* All operations inside `services.py`
* Models inside `models.py`
* Use functions, not classes (unless spec changes later)
